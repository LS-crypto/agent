1. 这份文档是什么

你不是从零写 Agent，而是在一个已经搭好的全栈项目上继续开发。 这个项目叫 Sheldon Agent：用阿里云百炼 Qwen 大模型 + Python 工具循环， 实现「能读文件、改代码、跑命令」的编程助手。

三个入口，一套核心：
  • apps/cli — 终端里直接对话
  • server + apps/web — 浏览器三栏 UI（会话 / 聊天 / 工具时间线）
  • core/ — CLI 和 Web 共用的 Agent、工具、Skills 逻辑
改 Agent 行为，通常只改 core/，CLI 和 Web 自动受益。

配套文档：

  • 开发日志.md — 已做了什么
  • 任务清单.md — 还要做什么
  • sheldon-agent/README.md — 快速上手命令
  • sheldon-agent/docs/deploy-ecs.md — 上云部署

3. Agent 核心原理

传统聊天：用户问 → 模型答 → 结束。
Agent（智能体）：用户问 → 模型决定要不要调工具 → 执行工具拿结果 → 再喂给模型 → 循环直到给出最终文字答案。

这叫 Agentic Loop（代理循环），是本项目的核心设计模式。

3.1 Agentic Loop — 一轮对话里发生了什么

代码在 core/agent/loop.py 的 AgentLoop.run()。

用户输入 "帮我在沙箱里创建 hello.py" │ ▼ ┌───────────────────┐ │ 1. 拼 messages │ system prompt + 历史 + 用户消息 +（可选）向量记忆 └─────────┬─────────┘ ▼ ┌───────────────────┐ │ 2. 选模型 │ auto → 路由选 flash/plus/max；固定模型 → 全程同一模型 └─────────┬─────────┘ ▼ ┌───────────────────┐ 没有 tool_calls │ 3. 调百炼 API │ ──────────────────────► 返回最终回复，结束 │ + 工具 Schema │ └─────────┬─────────┘ │ 有 tool_calls（如 write_file） ▼ ┌───────────────────┐ │ 4. 工具门禁 │ 速率限制 → 风险分级 → 用户确认（review 级） └─────────┬─────────┘ ▼ ┌───────────────────┐ │ 5. 执行工具 │ 在沙箱目录内读写文件 / 跑命令 └─────────┬─────────┘ ▼ ┌───────────────────┐ │ 6. 结果压缩 │ 太长则裁剪，省 token └─────────┬─────────┘ ▼ ┌───────────────────┐ │ 7. 结果写入 │ role=tool 的消息追加到 messages │ messages │ └─────────┬─────────┘ │ └──► 回到步骤 2（下一轮），最多 15 轮

关键文件调用链：

apps/cli/main.py 或 server/services/agent_service.py
    └── CodingAgent.chat()          # core/agent/coding_agent.py
            ├── resolve_model_choice()   # 固定模型 or 自动路由
            ├── VectorMemory           # 相关历史记忆注入
            └── AgentLoop.run()        # core/agent/loop.py
                    ├── route()        # 首轮复杂度路由
                    ├── client.chat.completions.create(tools=...)
                    └── run_tool_with_policy()  # core/agent/tool_gate.py
messages 是什么？ 就是发给大模型的对话数组，格式与 OpenAI Chat API 相同： system、user、assistant（可含 tool_calls）、tool。 每一轮工具执行后都会往里面追加，所以上下文会不断增长——所以有压缩器和会话重置。

3.2 工具系统 — 模型能「动手」的原因

工具 = 普通 Python 函数 + 一份 JSON Schema 描述。模型看到 Schema 后，用 tool_calls 告诉你「我要调哪个函数、参数是什么」。

工具名作用风险
read_file读沙箱内文件allowed
list_dir列目录allowed
grep / glob_search搜索代码allowed
write_file / edit_file写改文件review（需确认）
execute_command沙箱内跑 shellreview
git_status / git_diffGit 只读allowed
git_commit提交review

注册流程：

core/tools/build.py :: build_coding_registry()
    ├── register_filesystem_tools()   # 读写文件
    ├── register_search_tools()       # grep/glob
    ├── register_shell_tools()        # execute_command
    ├── register_git_tools()          # git_*
    └── register_mcp_tools()          # 可选 MCP 远端工具

每个工具在 core/tools/policy.py 里有风险等级：

  • allowed — 直接执行
  • review — 弹窗/终端问用户「允许吗？」
  • blocked — 拒绝（如读 .env、危险 shell 命令）

沙箱原理

用户的工作目录在 runtime/workspaces/{user_id}/projects/。 所有文件路径经 core/tools/sandbox.py 校验，禁止跳出沙箱（含 symlink 防护）。

3.3 模型与路由 — 像 Cursor 一样切换

目录在 core/models/catalog.py（静态择优）+ core/models/sync.py（可选远程校验账号是否开通）。

用户选择行为
auto（自动路由） 首轮用 core/agent/router.py 判断任务复杂度： simple → flash、standard → plus、complex → max。 后续轮根据上次工具类型再调（读文件用 flash，写文件用 plus）。
固定模型（如 qwen3.7-plus） 关闭路由，全程同一模型

路由两种手段（成本从低到高）：

  1. 关键词匹配 — 「列出」「搜索」→ simple；「重构」「架构」→ complex
  2. LLM 分类 — 关键词不确定时，用最便宜的 flash 模型问一句「simple/standard/complex」

3.4 安全与确认 — Human-in-the-Loop

Agent 能改文件、跑命令，必须有闸门口。

模型要调 write_file │ ▼ tool_gate.run_tool_with_policy() │ ├─ 速率限制（30次/分，100次/时） ├─ policy 判断 risk=review └─ confirm_handler() │ ├─ CLI：终端打印预览，问 y/N └─ Web：SSE 推 confirmation_required → 前端弹窗 → 用户点允许 → POST /api/chat/confirm → ConfirmationManager 唤醒等待中的 Agent 线程

活动日志经 core/agent/sanitize.py 脱敏后再写入 runtime/logs/，避免 API Key 泄露。

5. CLI 请求全流程

你在终端输入: uv run python -m apps.cli --model auto │ ▼ apps/cli/main.py · 解析 --model / --yes / --verbose · 创建 CodingAgent(user_id, resume=True) · 进入 while True: input("You> ") │ ▼ CodingAgent.chat(user_input, model=args.model) · build_skills_prompt() 注入技能说明到 system · resolve_model_choice("auto") → enable_routing=True · VectorMemory 检索相关片段 │ ▼ AgentLoop.run(enriched_input, messages) · 最多 15 轮 API 调用 · verbose=False 时终端只显示最终 Agent> 回复 · 详细过程写入 runtime/logs/{日期}/{user_id}.jsonl │ ▼ Session.save() → runtime/workspaces/{user}/sessions/*.json

CLI 适合做什么： 快速验证 Agent 逻辑、调试工具、看日志。改完 core/ 后第一时间用 CLI 测。

6. Web 请求全流程

浏览器 apps/web (Vite dev :5173) · 启动时 GET /health、GET /api/models、GET /api/sessions · 无会话则 POST /api/sessions 创建 │ │ 用户点发送 ▼ POST /api/chat (body: session_id, message, model) │ ▼ server/routes/chat.py → AgentService.stream_chat() · 从 SQLite 读出会话 messages · 新开线程跑 CodingAgent.chat()（不阻塞 async） · ActivityLogger.on_event → queue → SSE 推给前端 │ ▼ 前端 api/sse.ts 解析 data: {...}\n\n · loop_round / tool_call / tool_result / assistant_reply · confirmation_required → ConfirmDialog │ ▼ 流结束 event=done → 刷新会话列表，消息已写入 SQLite

SSE 事件类型（前端 ToolPanel 展示）

event含义
loop_round第 N 轮开始
tool_call模型决定调某工具
tool_result工具执行结果摘要
assistant_reply流式最终回复片段
confirmation_required需要用户确认
done本轮结束
error异常信息

开发时开两个终端：后端 uv run python -m server（默认 :8765），前端 cd apps/web && npm run dev。Vite 会把 /api 代理到后端。

8. Skills 技能库

Skills 不是可执行代码，而是写给模型的额外说明书（Markdown）， 在创建 Agent 时拼进 system prompt，让模型遵守特定规范（如 Karpathy 简洁原则、前端设计品味）。 

core/skills/loader.py :: discover_skills()
  扫描路径:
    · 工作区/taste-skill/.agents/skills/**/SKILL.md
    · 工作区/andrej-karpathy-skills/skills/**/SKILL.md
    · core/skills/custom/**/SKILL.md

build_skills_prompt(["karpathy-guidelines", "design-taste-frontend"])
  → 把指定技能的 SKILL.md 正文注入 system prompt

工作区里的 andrej-karpathy-skills/ 和 taste-skill/ 是只读引用，不要改坏；要加自己的技能可以放到 core/skills/custom/。

11. 常见问题

Q: 改了 core 但 Web 没变化？

重启 uv run python -m server。Python 后端不会热重载（除非你用 --reload 开发模式）。前端 Vite 会自动热更新。

Q: API 报错「请设置 DASHSCOPE_API_KEY」？

检查 sheldon-agent/.env 是否有 Key，并运行 uv run python scripts/check_env.py。

Q: Agent 一直调工具不结束？

看 runtime/logs/ 里每轮 tool_call；可能是任务太难或工具返回 error 导致反复重试。可调低 max_iterations 或改进 prompt。

Q: CLI 和 Web 会话互通吗？

不互通。 CLI 用 JSON 文件存会话，Web 用 SQLite。共用同一套 Agent 逻辑，但存储分离。

Q: 怎么调试 SSE？

浏览器 F12 → Network → 找 chat 请求 → EventStream 看 data: 行；或看右侧 ToolPanel 时间线。

Q: uv sync 失败？

网络/DNS 问题时可暂用 pip install 装 README 里列的依赖，但长期仍建议修好 uv。
